对于黑箱测试、白箱测试的理解

黑箱测试

黑箱测试指的是在不考虑程序内部逻辑和结构的情况下，通过输入一组合理的测试用例来验证程序输出是否符合预期。测试者只知道测试程序的输入输出，而不需要管内部的实现逻辑，也就是功能对了就ok。课程的中测和强测就体现了黑箱测试，通过编写测试数据进行正确性判断，只要输出和标准输出一致，说明程序就是正确的。但这里的正确是功能上的正确。

白箱测试

白箱测试指的是在考虑程序内部逻辑和结构的情况下，通过测试数据的输入来判断程序在执行过程中的各个部分是怎么运行的，验证数据在程序中的流动路径是怎么样的，并确定适当的输出确定程序的某一个模块的正确性。就好比一个程序处理需要经过多级函数的调用，白箱测试就是要检测每一个函数的处理结果是正确的，因为虽然可能程序通过了黑箱测试，但是实际上是由于一个函数出现了bug，在之后的函数中又出现了另外的bug，两个bug相互作用，导致最终结果却正确了，虽然总体的功能是正确的，但是给整个程序埋下了隐患，在新增功能的情况下，可能会出现比较大的bug，并且难以定位。

总的来说，我认为黑箱测试就是输入输出的正确性检测，白箱测试是输入输出加数据流动路径的正确性检测。相比之下，我认为白箱测试的覆盖度更高。

对单元测试、功能测试、集成测试、压力测试、回归测试的理解

单元测试

单元测试指的是对于程序的最小单元进行测试，在本课程中，单元测试可以认为是对于一个方法的测试，或者对于一个类的测试，但是事实上，如果一个类的方法是相互独立的话，对于只需要对于每一个方法进行充分测试即可。

功能测试

功能测试指的是对于程序实现功能正确性以及是否满足用户需求的一个检测，咱们的强测中测就体现的是功能测试。

集成测试

集成测试是指将多个模块结合起来，交互地进行测试。因为局部正确不代表全局正确，对于函数的调用，还有接口的使用还要遵守事先的约定，在作业中多次体现了集成化的设计，通过函数的相互调用实现顶层函数的功能。

压力测试

压力测试是指通过模拟实际应用的软硬件环境及用户使用过程的系统负荷，长时间或超大负荷地运行测试软件，来测试被测系统的性能、可靠性、稳定性等。在这一单元的作业中，课下测试所构造的极端数据，可能会导致超时的数据都是对于整个程序的压力测试，压力测试在作业中也是对于程序性能的测试。

回归测试

回归测试是指在迭代开发之后，使用原来已经通过测试的数据再次进行测试，以免因为新功能的加入导致原有功能的bug，同时也是对于新增功能的正确性检测。就像每次作业迭代时，我们会用上一次强测的数据对于此次作业做一个初步的测试。

是否使用了测试工具

在这一单元的课下测试中，我使用python实现了一个数据生成与构造的对拍器。主要使用逻辑是使用两个人的jar包进行对拍，通过测试数据生成和对拍一体化流程进行正确性的判定，如果如果两方对于数据的结果都不确定，通过寻找其他人进行对拍求证正确答案。以下是测评机的大致框架：

# --------------------config-----------------------
n = 100  # 设置测试点总数
generator = 'code.jar'  # 用于数据生成的jar包
tester = 'mine.jar'  # 用于对拍的jar包
file_add = "data"
echo_on = True  # 是否显示输出信息
# -------------------------------------------------
print("======DATA GENERATING======")
print(f'Expected waiting time {2 * n}s.')
os.system(f'python generator.py {n} {generator}')
print("=========COMPLETE==========")
folder = os.path.exists('out')
if not folder:
    os.makedirs('out')
folder = os.path.exists('diff')
if not folder:
    os.makedirs('diff')
for home, dirs, files in os.walk(file_add):
    for i in range(1, len(files) + 1):
        os.system(f'java -jar {tester}<data/random{i}.txt >out/out{i}.txt')
        print(f'testId:{i}')
        if not echo_on:
            os.system(f'fc ans/ans{i}.txt out/out{i}.txt >diff/diff{i}.txt')
        else:
            ok = cmp_file(f'ans/ans{i}.txt', f'out/out{i}.txt')
            if ok:
                pass_cnt = pass_cnt + 1
if pass_cnt == n:
    print("You have passed all the tests.")
else:
    print(f"You have {n - pass_cnt} errors.")
print("========TEST FINISH========")

数据构造有何策略

在三次作业的数据构造方面，不仅要注重数据的全面性，但同时也要注重压力测试，以应对极端数据的情况。在三次作业中，大部分简单方法都可以通过直接翻译jml的描述实现，其算法复杂度大多是在o(n)，对于10000条数据所构成的图性能影响不是特别大，这时，就需要着重关注功能实现的正确性而降低复杂度的关注度，这些方法的测试就可以通过评测机构造的大量随机数据来测试。

但是，对于jml描述中本就是o(n2)或者o(n3)复杂度的方法，正确性是一个要求，但是在要求限时的情况下的就要求对于时间性能的考量，在每一次作业中，这些方法的正确性测试也可以通过评测机构造的随机数据进行测试。除此以外，我对于每次作业中出现的高复杂度的数据都进行了压力测试：

在第九次作业中，queryTripleSum和queryBlockSum方法的复杂度较高，因此我构造了加一个人或加一个关系就调用一次queryTripleSum和queryBlockSum方法，用于测试复杂度。

在第十次作业中，queryCoupleSum和qgvs方法的复杂度较高，并且实际上维护起来的代价也不低，我构造了一个三千人的大关系网，在添加关系的过程中，反复调用queryCoupleSum，完成关系构建之后，连续调用多次qgvs，来测试qgvs动态维护的效果，经过测试，每次计算qgvs的策略会导致运行时间超长可能最终导致强测TLE。

在第十一次作业中，queryLeastMoments方法的复杂度很高，于是我构造了一些完全图，通过将所有的关系都进行构建，并且在增加关系时也调用queryLeastMoments方法，最终测试结果发现，只有使用了复杂度较低，即只需要跑一次迪杰斯特拉算法的实现方法才能有效通过测试，否则会导致很长时间结果还不能算出来。

架构设计与图模型构建和维护策略

第九次作业

并查集

在这一次的关系网构建中，我使用了并查集的方法，对于若干组可以相互联系到的人构建出相应的并查集网络，这样在查询两个人的关系时，只需要查询他们是否有公共的根节点就可判断他们是否可以联系到。

构建并查集

那么如何构建并查集呢，我们需要对于每一个人维护一个父亲结点，在初始化时，需要把父节点指向自己，之后如果添加两人的关系，需要使用find方法先找到两人所在的并查集的根节点，之后，选择一个根节点作为另一组并查集根节点的父节点，这样，就完成了并查集的合并，之后我们判断两人是否同初一个并查集时，只需调用find函数查找他们的根节点是否相同，即可实现相应判断。

int find(int x)
{
    if(fa[x] == x)
        return x;
    else
        return find(fa[x]);
}

动态维护

在第九次的作业中，有两个jml是高复杂度方法，queryBlockSum()，queryTripleSum()，分别是用来求整个关系网中的分支数量，以及整个关系网中三个人互相认识的关系数量，如果使用jml直接翻译过来的方法，在反复调用方法的情况下会导致时间的极大性能损失，因此我们需要引入动态维护机制。

对于BlockSum方法，当我们新增一个人时，blockSum的数量就加一，因为单独的人就是一个分支，接下来，如果我们要添加两个人的关系，那么他们对应的两个并查集就需要合并，在并查集合并时，如果原本两个人处于不同的并查集，那么合并后总的分支数量会减一，如果两个人就处于同一个并查集，那么blockSum的数量就不变。

对于queryTripleSum方法，当我们两人关系时，如果两人拥有相同的熟人，那么就会增加一个三角形，因此，我们在增加关系时，只需遍历一个人的熟人，如果这个熟人和增加关系的另一方有联系，那么tripleSum就可以加一。

那么最终，调用方法时，我们只需返回动态维护的值即可，方法的复杂度为O(1)。

public int queryBlockSum() {
    return blockSum;
}

public int queryTripleSum() {
    return tripleSum;
}

第十次作业

第十次作业在上一次作业的基础上出现图中两人关系的删除，但是对于并查集来说，增加关系简单，删除关系有些困难，因此如果删关系，如果删除后两人还处在一个并查集，不用做任何处理，但是如果两个的关系删除导致两个人分别出于两个并查集，就需要分别更新两个并查集中人的根节点，简单起见，可以将两个拆开的并查集的根节点分别设置为删除关系的两个人，这样既维护了并查集，也保证了之后并查集使用的正确性。

由于删除关系后原先的并查集被破坏，我们只能使用bfs检查被删除关系的两个结点是否仍然可达，如果可达就不需要做任何操作，如果不可达就重新构建两个新的并查集，相应的此时blockSum会加一。

动态维护

在这一次作业中，由于删边的存在，需要动态维护blockSum还有tripleSum，前面已经说了blocksum的维护方法，tripleSum的维护方法也很简单，和动态添加tripleSum的操作相反，当两个人的关系删除时，遍历他们的相同熟人，每遇到一个相同熟人，tripleSum就会减一。

动态维护——用时计算机制

在这一次的作业中，有一个qgvs的方法，按照jml的写法有o(n3)的复杂度，如果每调用一次计算一次会导致性能大量损失，但是这个地方动态维护的代价也不小，因此我就想仿照OS中写时复制的机制，使用一个变量存储qgvs的结果，如果结果没有被修改，调用方法时可以直接被返回，如果结果可能被修改时，我设置了一个dirty位用于标记结果被修改，直到下一次调用方法时才会被重新计算并把脏位恢复，这样就减少了计算的代价。

当往一个组中加人，删人，对于两个人之间的value修改时都会修改脏位，因为此时qgvs的结果应该发生变化，在其他场景下不会改变组内的qgvs值。

//MyGroup.java
public void addPerson(Person person) {
        people.put(person.getId(), person);
        modifiedValue = true;
}
public void deletePerson(Person person) {
        people.remove(person.getId(), person);
        modifiedValue = true;
}

第十一次作业

在最后一次作业中，对于之前需要动态维护的方法都没有改动，对于图的构建也没有变化，只是增加了qlm的方法，要求找到经过一个人的权值最小的环的权重。我曾思考过对于这个进行动态维护，但是由于整个图中的人很多，加人和加关系可能会导致多个人的环需要重新构建，动态维护可能被单次计算的代价还大，违背了降低复杂度的初衷，因此，我使用了讨论区中的优先队列+魔改dijkstra的方法，通过调用一次dijkstra的方法实现最小权值的计算。

规格与实现分离及性能问题

规格和实现分离

通过三次作业的练习，我意识到规格是对于我们的功能实现给了一个最基本baseline，即简单机械翻译jml能保证我们代码和功能的正确性，但是仅仅是正确性，对于实现的优雅性是不太保证的，即我们只能得到一个正确性的标程，但是代码的效率方面可能就会”惨不忍睹“。因此在这三次的作业中，我们需要从jml中知道我们需要做什么，然后以这个需求为驱动去实现较高效率的代码的开发，对于我们来说，对于需求的实现是自由的，满足规范即可。

实现中的性能问题和修改方式

在这一单元的作业中，我没有出现性能问题，主要得益于同学们之间的相互讨论以及大佬们对于高效方法的慷慨分享。除此以外，课下的测试数据构造也是发现性能问题的好方法，在某次作业中，我使用自己构造的数据让程序跑不动了，我就知道需要换一种性能更高的方法进行功能的实现。

简单总结，三次作业对于性能问题的修改方法有三个方面：

第一个方面是动态维护，调用查询方法的时只需返回已经计算好并且已事先储存的元素，复杂度极地，相应的复杂度被分散在动态维护时，但大多数是o(n)的复杂度。

第二个方面是引入一个类似于写时复制的机制，引入一个dirty位，当修改值时，先不着急计算，等到下一次要查询值再进行计算，虽然一次算法的复杂度可能较高，但是在不修改的情况下，可以直接返回原来存储的值，算法的复杂度还是大大降低了。

第三个方面是真正降低算法的复杂度，寻求更高效的算法。前面两个方面是间接减少复杂度，而这个就是在算法上下功夫，对于算法还是提出了很高的要求，需要对于算法有着很深的理解，但是一旦想到就是可以称作神一样的实现。

OkTest

本单元的oktest在我看来就是对于就是对于方法的白盒测试，在测试的过程中，我们对于方法的前置条件，后置条件，以及调用方法所得到的结果都进行了检测，以保证方法的完整意义上的正确性，同时oktest的每一个检测点也相应检查了方法是否满足对应的规格。对于这个的建议是可以将oktest放在一个静态类中，这样感觉更符合面向对象的思想，因为测试应该要和实现功能分离。

学习体会

通过本单元的学习，我学习到了规格化设计的精妙之处，学到了竟然还有规范化的描述语言来指导需求的实现，学习到了规格与设计相分离的思想，感受到了从顶层到底层的设计延伸。但同时我也体会到了编写jml语言的困难，在实验中，感觉对于实现jml语言的简洁性和正确性都存在一定的困难。在之后的学习中，我会以这种规格化设计的思想指导设计，通过本单元学到的测试方法，对于未来的项目开发做到全面且高效的测试。
